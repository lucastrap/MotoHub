C2.1.2 Système d'intégration continue A2.1

Configurer l'intégration continue : fusionner les codes sources et tester régulièrement les blocs de code pour réduire les risques de régression.

▹ Protocole d'intégration & de déploiement continus

Le pipeline .github/workflows/ci.yml s'exécute sur chaque push et chaque pull request vers main et develop. Il enchaîne des séquences ordonnées et conditionnelles :

push / PR ──▶ Job « test-and-build » (Ubuntu + service PostgreSQL 15)
   1. checkout du code
   2. npm ci                     (installation reproductible)
   3. prisma generate + migrate  (préparation base éphémère)
   4. next lint                  (qualité)
   5. npm audit --audit-level=high  (veille vulnérabilités — OWASP A06)
   6. jest                       (tests unitaires + couverture)
      └─ upload du rapport de couverture (artefact)
   7. next build                 (compilation production)
   8. playwright test            (tests end-to-end)
   └─ upload du rapport Playwright (artefact)
        │
        ▼  si branche = main ET event = push ET job précédent vert
   Job « deploy » ──▶ vercel --prod   (déploiement continu)
Réduction des régressions
Aucun code n'atteint la production sans avoir franchi lint + 99 tests unitaires + build + 22 tests e2e. La fusion d'une PR est conditionnée à la réussite de ces séquences.

C2.2.1 Prototype & architecture logicielle Éliminatoire

Concevoir un prototype fonctionnel, structuré et maintenable, tenant compte de l'ergonomie, des équipements ciblés et des exigences de sécurité.

▹ Architecture & maintenabilité

L'application suit une architecture en couches favorisant la maintenabilité et la séparation des responsabilités :

CoucheEmplacementResponsabilité
Présentationsrc/app/**/page.tsx, src/componentsPages et composants UI réutilisables
API / métiersrc/app/api/**/route.tsEndpoints REST, règles métier
Sécurité transversesrc/middleware.ts, src/lib/auth.tsProtection des routes, JWT
Accès aux donnéessrc/lib/prisma.ts, prisma/schema.prismaORM, modèle de données
Utilitairessrc/lib/*Formatage, logger, helpers

▹ Frameworks & paradigmes

  • Next.js 14 App Router — rendu serveur, routage par convention de fichiers.
  • Programmation par composants React (composition, forwardRef).
  • Validation déclarative avec Zod (schémas typés réutilisés côté client et serveur).
  • Design system (Tailwind + variantes cva) pour la cohérence UI.

Cette organisation applique le principe de séparation des préoccupations : une modification de l'interface n'impacte pas les règles métier, et une évolution de la base de données reste confinée à la couche d'accès aux données via Prisma. Le typage statique strict de TypeScript, activé sur l'ensemble du projet (tsc --noEmit sans erreur), constitue un premier filet de sécurité pour la maintenabilité : toute rupture de contrat entre couches est détectée à la compilation. Les schémas Zod sont partagés entre le client (React Hook Form) et le serveur (routes API), ce qui évite la duplication des règles de validation et garantit leur cohérence — un gage de maintenabilité important.

▹ Fonctionnalités principales & user stories

Le prototype couvre un ensemble cohérent de fonctionnalités traduisant directement les user stories exprimées par le commanditaire. Chacune est fonctionnelle et manipulable de bout en bout :

« … gérer plusieurs motos »Garage multi-motos, wizard 3 étapes, moto principaleAssistant multi-étapes, cartes, badges, menus
« … suivre l'entretien et ne rater aucune échéance »Carnet d'entretien, MàJ auto du kilométrage, échéances calculéesFormulaire, listes filtrables, tableau de bord

L'ergonomie tient compte des équipements ciblés : interface responsive web et application installable (PWA) sur mobile, navigation adaptée (menu latéral sur desktop, barre défilante sur mobile). Les exigences de sécurité sont intégrées dès la conception (défense en profondeur détaillée en C2.2.3).

C2.2.2 Harnais de tests unitaires Éliminatoire

Développer un harnais de tests unitaires pour prévenir les régressions et garantir le bon fonctionnement du logiciel.

▹ Stratégie de test & interprétation du critère

Le critère éliminatoire exige que « les tests unitaires couvrent la majorité du code développé ». Ce critère mérite une lecture rigoureuse : dans une application Next.js, tout le code n'est pas de même nature. On distingue une couche logique (librairies, routes API, middleware, validation, composants) — testable en isolation par des tests unitaires — et une couche de présentation lourde (pages intégrant appels réseau, rendu 3D, état complexe) qui relève par nature du test end-to-end. La stratégie retenue est donc explicitement à deux niveaux :

  • Tests unitaires (Jest) — 142 tests ciblant la couche logique et les parcours critiques (authentification, navigation), là où se concentrent les règles métier et la sécurité.
  • Tests end-to-end (Playwright) — 22 tests validant les pages et les parcours utilisateur réels (voir C2.3.1).

Le périmètre de mesure unitaire (collectCoverageFrom dans jest.config.js) est documenté et assumé, avec des seuils bloquants en CI (statements 90 %, branches 80 %, functions 90 %, lines 90 %). Cette configuration est ouverte à la vérification du jury.

▹ Transparence sur la couverture globale

Par honnêteté intellectuelle, deux chiffres sont présentés. Sur le périmètre ciblé (logique + parcours critiques), la couverture atteint 96 % des instructions. Sur l'intégralité du dossier src/ (pages lourdes comprises), elle s'établit désormais à ≈ 55 % des instructions (55 % des lignes) — soit une majorité du code développé, ce qui satisfait le critère au sens strict. Ce résultat a été obtenu en refactorisant la logique des pages complexes (ex. le wizard garage/new) dans des modules purs et testables. Le reliquat non couvert unitairement correspond au JSX de présentation, validé par les tests e2e : la combinaison unitaire + e2e couvre donc l'essentiel du code.

▹ Accessibilité — référentiel RGAA

Le RGAA (Référentiel Général d'Amélioration de l'Accessibilité) a été retenu car il constitue la norme officielle française, alignée sur les WCAG 2.1, adaptée à un service à destination d'une association. Actions mises en œuvre :

Critère RGAAAction
8.3 — Langue<html lang="fr">
9.1 — Structure de titresUn <h1> unique par page
11.1 / 11.13 — FormulairesÉtiquettes <Label htmlFor> + autocomplete
11.10 — Erreurs de saisierole="alert", aria-invalid, aria-describedby
12.7 — Lien d'évitement« Aller au contenu principal » vers #main-content
1.2 — Images décorativesIcônes aria-hidden
12.2 — Navigation<nav aria-label> + aria-current="page"

Le choix du RGAA plutôt qu'OPQUAST est justifié : OPQUAST est une check-list de qualité web généraliste, tandis que le RGAA est spécifiquement dédié à l'accessibilité numérique, opposable et aligné sur les WCAG 2.1 — plus pertinent pour démontrer la prise en compte des personnes en situation de handicap. Les correctifs ont été appliqués sur les parcours critiques (authentification, navigation principale) et sont prouvés par des tests automatisés. Une grille d'audit chiffrée (docs/accessibilite-rgaa.md) évalue 18 critères RGAA sur ce périmètre : 100 % de conformité sur les 17 critères applicables.

Accessibilité testée automatiquement
5 tests e2e dédiés (tests/e2e/accessibility.spec.ts) vérifient langue, étiquettes, autocomplétion, signalement d'erreurs (role=alert / aria-invalid) et structure de titres — la conformité est donc vérifiable, non déclarative.

A Annexe A — Glossaire technique

Définition des principaux termes techniques employés dans ce rapport.

TermeDéfinition
API RoutePoint d'entrée serveur de Next.js (fichier route.ts) exposant une fonction HTTP (GET, POST…) ; contient les règles métier côté serveur.
App RouterSystème de routage de Next.js 14 fondé sur l'arborescence de fichiers du dossier app/, avec rendu serveur par défaut.
bcryptAlgorithme de hachage de mots de passe avec « sel » et facteur de coût, résistant aux attaques par force brute.
CI/CDIntégration Continue / Déploiement Continu : automatisation de la vérification (tests, build) et de la mise en production à chaque modification du code.
Cookie httpOnlyCookie inaccessible au JavaScript du navigateur, protégeant le jeton de session contre le vol par script (XSS).
Couverture (coverage)Pourcentage du code exécuté par les tests. Mesurée en statements (instructions), branches, functions et lines.
cvaclass-variance-authority : utilitaire de gestion des variantes de style d'un composant (taille, couleur…).
e2e (end-to-end)Test de bout en bout simulant un utilisateur réel dans un navigateur, du clic au résultat affiché.
ESLintAnalyseur statique de code JavaScript/TypeScript appliquant des règles de qualité et de bonnes pratiques.
JestFramework de tests unitaires JavaScript, ici couplé à Testing Library pour le rendu de composants React.
JWTJSON Web Token : jeton signé encodant l'identité de l'utilisateur, vérifié à chaque requête pour authentifier la session.
joseLibrairie de création et vérification de jetons JWT compatible avec l'environnement Edge de Next.js.
LighthouseOutil d'audit intégré à Chrome mesurant performance, accessibilité, bonnes pratiques et SEO d'une page.
MiddlewareFonction exécutée avant l'accès à une route ; ici, filtre d'authentification protégeant les pages privées.
MigrationScript versionné décrivant une évolution du schéma de base de données, rejouable de façon reproductible (Prisma).
MockObjet de substitution simulant une dépendance (base de données, API externe) pour isoler le code testé.
NHTSANational Highway Traffic Safety Administration : API publique fournissant la liste officielle des modèles de véhicules.
ORMObject-Relational Mapping : couche traduisant les objets du code en requêtes SQL (ici Prisma), avec requêtes paramétrées.
OWASPOpen Worldwide Application Security Project : organisation de référence en sécurité applicative, éditrice du « Top 10 » des risques.
PlaywrightFramework de tests end-to-end multi-navigateur, avec rapport HTML et exécution en CI.
PrismaORM TypeScript pour PostgreSQL : schéma déclaratif, migrations, client typé.
PWAProgressive Web App : application web installable sur mobile, utilisable comme une application native.
RGAARéférentiel Général d'Amélioration de l'Accessibilité : norme française d'accessibilité numérique, alignée sur les WCAG.
RollbackRetour à une version antérieure stable de l'application en cas d'anomalie en production.
SIVSystème d'Immatriculation des Véhicules : format français des plaques (AB-123-CD).
SSRServer-Side Rendering : génération du HTML côté serveur avant envoi au navigateur (meilleur SEO et temps d'affichage).
Tailwind CSSFramework CSS « utilitaire » : la mise en forme s'exprime via des classes prédéfinies dans le balisage.
Testing LibraryBibliothèque de test de composants React centrée sur l'usage réel (rôles, libellés) plutôt que sur l'implémentation.
TypeScriptSur-ensemble typé de JavaScript ; le typage statique détecte des erreurs dès la compilation.
VercelPlateforme d'hébergement et de déploiement continu optimisée pour Next.js.
WinstonBibliothèque de journalisation structurée (niveaux, horodatage, format JSON).
ZodBibliothèque de validation de schémas TypeScript ; garantit la conformité des données entrantes.
